# 求职申请管理看板 + AI 求职助手

一个面向中文求职场景的单页 Web 应用，用来管理岗位申请进度，并结合 OpenAI 提供 AI 求职建议。它既能作为申请追踪工具，也能作为简历优化、面试准备、投递策略整理的辅助工作台。

---

## 一、项目定位

本项目服务于求职过程中的两个核心问题：

1. 申请信息分散，容易遗漏 deadline、面试安排和材料准备情况。
2. 用户虽然记录了岗位信息，但缺少一个能结合真实上下文提供建议的 AI 助手。

因此，项目将「岗位管理」与「AI 咨询」合并到同一个页面中：

- 用看板和列表管理申请生命周期
- 用结构化字段沉淀岗位信息、JD、材料和面试记录
- 用 AI 结合当前申请数据、岗位详情和上传的简历文件提供建议

---

## 二、技术架构

### 1. 整体架构

项目采用轻量前后端分离但同仓部署的方式：

- 前端：单文件 `index.html`
- 后端：Node.js + Express `server.js`
- AI：通过 OpenAI SDK 调用大模型
- 存储：浏览器 `localStorage`

整体数据流如下：

1. 用户在前端录入岗位、JD、备注、材料、面试记录
2. 前端将岗位数据持久化到浏览器本地
3. 用户打开 AI 助手发起提问时：
   - 前端整理申请概况
   - 前端整理岗位详情上下文
   - 如果有上传简历/图片，也会先在前端处理后一起发送
4. 后端将这些上下文拼接为系统提示词并转发给 OpenAI
5. 模型以流式方式返回内容，前端实时渲染聊天结果

### 2. 前端技术栈

- React 18 UMD
- ReactDOM 18 UMD
- Babel Standalone
- Tailwind CSS CDN
- Marked + DOMPurify
- Mammoth：解析 `.docx`
- PDF.js：解析 `.pdf`

前端特点：

- 无打包器、无构建步骤，适合快速原型和本地运行
- UI、状态、业务逻辑集中在一个 `index.html` 中
- AI 对话支持 Markdown 渲染与流式输出

### 3. 后端技术栈

- Node.js 20+
- Express 4
- dotenv
- OpenAI Node SDK

后端职责：

- 提供静态资源访问
- 暴露 `/api/health` 健康检查接口
- 暴露 `/api/chat` AI 流式聊天接口
- 拼装岗位上下文、申请概况和用户消息
- 将模型返回内容通过 SSE 推送给前端

### 4. 数据存储设计

当前项目没有数据库，所有岗位数据保存在浏览器本地：

- 存储方式：`localStorage`
- 键名：`jobApplications.v1`

优势：

- 零部署数据库成本
- 本地体验快
- 适合个人使用和原型验证

局限：

- 无多端同步
- 更换浏览器或清空缓存后数据会丢失
- 不适合多人协作场景

### 5. AI 上下文策略

当前 AI 助手不是只看用户一句问题，而是会综合以下内容：

- 申请概况：
  - 总申请数
  - 阶段分布
  - 7 天内截止事项
  - 当前面试中岗位
  - 当前 Offer
- 岗位详情：
  - 公司
  - 岗位名
  - 岗位类型
  - 当前阶段
  - deadline
  - 材料准备情况
  - 面试记录
  - 备注
  - JD 摘要
- 用户上传附件：
  - PDF / DOCX / TXT / Markdown 文本内容
  - 图片类简历或截图

这样 AI 给出的建议会更贴近真实申请状态，而不是通用答案。

## 三、产品设计

### 1. 产品目标

产品目标不是做一个单纯的职位表格，而是做一个「求职过程操作台」。

围绕用户的真实使用路径，设计上强调四件事：

1. 看得见全局：当前投了多少、卡在哪一轮、哪些快到期
2. 记得住细节：每个岗位的 JD、备注、面试记录、材料状态都能沉淀
3. 能快速行动：新增、编辑、拖拽阶段切换、快捷提问都很直接
4. 有智能辅助：AI 能结合真实数据给建议，不只是闲聊

### 2. 信息架构

页面主要由四部分组成：

- 顶部概览区
  - 标题
  - 总申请数 / 面试中 / Offer / 即将到期
  - 最近 deadline 提醒
- 申请管理区
  - Kanban 看板视图
  - 列表视图
- 岗位详情弹窗
  - 基础信息
  - JD
  - deadline
  - 材料
  - 面试记录
  - 备注
- AI 助手区
  - 快捷问题
  - 流式聊天
  - 附件上传

### 3. 交互设计思路

#### 看板视图

适合快速判断当前流程分布，强调过程感和节奏感：

- 每个阶段一列
- 支持拖拽卡片切换状态
- 支持从某一列快速新增岗位

#### 列表视图

适合手机和精细查阅：

- 支持阶段筛选
- 支持按 deadline / 投递日期排序

#### 详情弹窗

一个岗位的完整信息尽量在一个面板内完成编辑，减少跳转：

- 基础字段
- JD 文本
- deadline 列表
- 材料清单
- 面试记录
- 备注

#### AI 助手

设计目标是“像对着一个了解你现状的求职顾问提问”：

- 打开即可看到快捷问题
- 支持边生成边展示
- 支持上传简历和截图，减少复制粘贴

### 4. 视觉风格

产品视觉是轻量、克制、偏职业化的方向：

- 浅色背景
- 中性灰 + 少量状态色
- 卡片式布局
- 强调信息密度与清晰度
- 避免复杂装饰和过强的营销感

## 六、使用说明

### 1. 环境要求

- Node.js >= 20
- 一个可用的 OpenAI API Key

### 2. 安装依赖

```bash
npm install
```

### 3. 配置环境变量

复制 `.env.example` 为 `.env`，然后填写：

```env
OPENAI_API_KEY=你的_API_KEY
OPENAI_MODEL=gpt-5.4-mini
# OPENAI_BASE_URL=https://your-gateway.example.com/v1
PORT=3000
```

说明：

- `OPENAI_API_KEY`：必填
- `OPENAI_MODEL`：必填
- `OPENAI_BASE_URL`：可选，如果使用兼容网关可配置
- `PORT`：可选，默认 `3000`

### 4. 启动项目

开发模式：

```bash
npm run dev
```

普通启动：

```bash
npm start
```

启动后访问：

```text
http://localhost:3000
```

### 5. 基础使用流程

#### 新增岗位

1. 点击页面右上角“新增申请”
2. 填写公司、岗位、类型、阶段
3. 可选填写岗位 JD
4. 保存后进入看板或列表统一管理

#### 编辑岗位

1. 点击任意岗位卡片或列表行
2. 在详情弹窗中更新：
   - JD
   - deadline
   - 材料
   - 面试记录
   - 备注

#### 使用 AI 助手

1. 点击右下角 AI 按钮
2. 直接提问，或点击快捷问题
3. 也可以先上传简历或图片，再发送问题

推荐提问示例：

- 请根据我当前所有申请，帮我排本周优先级
- 结合这个岗位 JD，帮我改简历描述
- 根据我上传的简历，指出最影响通过率的问题
- 我现在材料准备还缺什么
- 根据面试记录，帮我判断下一轮要重点补什么

#### 上传附件

聊天框支持上传：

- PDF 简历
- DOCX 简历
- 文字版自我介绍
- 岗位截图
- 项目材料截图

上传后可以这样问：

- 请帮我优化这份简历
- 请结合我的申请数据，判断这份简历最适合投哪些岗位
- 请从产品经理实习角度重写这份简历
- 请提炼这张岗位截图里的核心要求

## 七、接口说明

### 1. `GET /api/health`

返回 AI 配置状态：

```json
{
  "configured": true
}
```

### 2. `POST /api/chat`

用于 AI 对话，前端会传递：

- 聊天历史
- 申请概况
- 岗位详情上下文
- 附件内容

后端会以 SSE 形式流式返回：

```text
data: {"delta":"你好"}

data: {"delta":"我来帮你看看"}

data: [DONE]
```

## 八、当前实现特点

### 优点

- 轻量，启动成本低
- 不依赖数据库，适合本地使用
- AI 建议不是空泛回答，而是结合真实岗位信息
- 支持简历文件和图片上传，使用门槛低
- 交互完整，适合作为求职产品原型

### 当前边界

- 数据只存在本地浏览器，没有账号体系
- 没有服务端持久化
- 没有多人协作和云端同步
- 大文件上传仍受浏览器和请求体大小限制
- 扫描版 PDF 的文本提取效果可能有限

## 十、注意事项

- 本项目中的岗位数据保存在本地浏览器，清空缓存会丢失数据
- `.env` 中的 API Key 不应提交到公开仓库
- 如果 AI 助手不可用，优先检查：
  - `.env` 是否配置正确
  - 后端是否已重启
  - 模型名是否可用
  - 网络是否能访问上游接口
